# Numazu Garden OS - 開発経緯・設計判断の記録

## プロジェクト概要

- 開始日: 2026-05-04
- 目的: 沼津市の気象特性に合わせた家庭菜園管理CLIツール
- 管理植物: ステラミニ（固定種トマト）、ストロング系ラベンダー

---

## 設計判断ログ

### 1. 外部API vs スタブ設計

**問題**: 天気APIは外部依存があり、APIキー管理・障害・コスト等のリスクがある。

**判断**: `fetch_weather()` をスタブ設計にし、外部API障害時は気象庁統計の平年値にフォールバック。

```python
def fetch_weather():
    try:
        # 将来API実装（今はNotImplementedErrorを投げる）
        raise NotImplementedError(...)
    except Exception:
        return get_weather_stub()  # 平年値スタブで継続
```

**結果**: API未設定でも即座に動作確認できる。ループが止まらない最重要ポイント。

---

### 2. データ構造: JSON vs SQLite

**問題**: 植物データをどこで管理するか。

**判断**: JSONファイル（`plants.json`）を選択。

**理由**:
- SQLiteはセットアップ不要だが、人間が読み書きしづらい
- 植物の種類は少数（数十件が上限）でRDBMSの恩恵が薄い
- JSONはgitで差分管理しやすく、手動編集も容易
- 将来的にSQLiteやTinyDBへの移行も容易

---

### 3. 沼津市特有の気象リスク対応

**観察**: 沼津市は以下の特有リスクを持つ。
- 台風（7〜9月）: 静岡県は台風の直撃・接近が全国最多級
- 梅雨（6〜7月）: 高温多湿でラベンダーの蒸れが深刻
- 夏の日照: 熱帯夜の発生でトマトの着果率が低下

**対応**: `advisor.py` で月・台風リスクをトリガーにしたアドバイス分岐を実装。
- `typhoon_risk: "high"` → 支柱補強・水やり調整アドバイスを追加
- 6月 + ラベンダー → 梅雨前剪定の緊急アドバイスを強調

---

### 4. 固定種（ヘアルーム）トマトの採種サポート

**背景**: ステラミニは固定種のため、毎年採種すれば種を購入せずに継続できる。

**実装**: `plants.json` に `seed_saving` セクションを追加。
- 採種方法・保管方法・発芽年限を記録
- `advisor.py` の生育ステージ判定で「採種期」を識別

---

### 5. CLI設計のシンプルさ優先

**判断**: `argparse` のみ使用。`click` や `rich` 等のサードパーティライブラリは使用しない。

**理由**:
- `requirements.txt` をほぼ空にすることで、どの環境でも即座に動く
- インストール手順がゼロに近いほど使用障壁が下がる
- 家庭菜園管理ツールに高度なUIは不要

## 開発プロセスのループと脱出

### ループ1: 「沼津の気象データをどこから取るか」

**ループ状態**: 気象庁APIを調べる → レート制限・認証要件を確認 → OpenWeatherMapを調べる → APIキー登録が必要 → 別のAPIを調べる...

**脱出判断**: 「外部APIが使えなくてもとにかく止まらずに動くもの」という要件を再確認。
1991-2020平年値を直接コードに埋め込む方針に切り替え。

**教訓**: 「完璧なデータ」より「今すぐ動くもの」を優先。スタブは後で差し替えればよい。

---

### ループ2: 「植物データの粒度」

**ループ状態**: 詳細なデータを書く → 水やりの条件が複雑すぎる → アドバイスロジックと重複 → どちらに書くべきか...

**脱出判断**: JSONには「静的な事実」（品種情報・スケジュール・ノート）だけを置き、「動的な判断」はadvisor.pyに集中させる分離方針を決定。

**教訓**: データ層とロジック層の責務分離を早期に決めるとループが短くなる。

---

### ループ3: 「ラベンダーの品種特性」

**ループ状態**: 「ストロング系」の定義を調べる → ラバンジン系との違い → 耐暑性の文献を確認 → 沼津の夏に耐えられるか...

**脱出判断**: 「ストロング系はラバンジン系の耐暑性品種群」という定義で確定。完璧な文献考証より、実際の管理アドバイス（梅雨前剪定・蒸れ対策）の精度を上げることに集中。

**教訓**: 品種論争より実践的な管理アドバイスの充実度が重要。

## v0.3 追加: Threads広報部 & 設定Webサーバー（2026-05-04）

### 設計思想: 「API後回し」の疎結合設計

**なぜWeb設定画面を先に作ったか**

APIキー（Threads Token）は機密情報であり、コードに直書きしてはならない。
かつ、APIキーが手元にない段階からシステムを動かし続けることが要件だった。

解決策として「設定チャンネル（config_server.py）とロジック（social_handler.py）を完全に分離」した。
具体的には:

```
.env ← config_server.py が書く
  ↓
social_handler.py が読む
  ↓
PLACEHOLDER → シミュレーション / 実トークン → 本番送信
```

この構造により:
- APIキーを持っていない今日でも `post` コマンドが動く
- 将来トークンを入手したら `.env` に書くだけで本番送信が始まる
- コードを一行も変えずに「後付けでAPIを繋ぐ」ことができる

---

### ループ4: 「Flaskを使うか標準ライブラリで実装するか」

**ループ状態**: Flask のほうが楽 → でもインストールが必要 → requirements.txt の方針に反する → 他のマイクロフレームワークを探す...

**脱出判断**: `http.server.BaseHTTPRequestHandler` で十分。フォームのPOST受信・HTMLレンダリング・リダイレクトの3つしか使わない。Flaskを入れる理由がない。

**教訓**: 標準ライブラリで完結できる範囲ではサードパーティを入れない。「将来スケールするかも」という憶測で今を複雑にしない。

---

### ループ5: 「Threads APIの認証フローをどこまで実装するか」

**ループ状態**: OAuth 2.0フローを調べる → ユーザーID取得が必要 → コールバックURLが必要 → ローカル開発環境でリダイレクト受信が難しい → 全部今実装するべきか...

**脱出判断**: 「APIが厳しければスタブ」という大原則に立ち戻る。
`_post_to_threads_api()` を `NotImplementedError` でラップし、呼び出し側（`post_text()`）が例外を捕捉してシミュレーションに切り替える設計を採用。
コード内に将来の実装例をコメントアウトで残し、差し替えの場所を明示した。

**教訓**: 実装が難しい部分はスタブで「場所を確保」してから前進する。NotImplementedError はドキュメントであり、後継開発者（自分含む）への道標でもある。

---

### セキュリティ設計: .env と .gitignore の関係

`.env` は APIトークンを含む機密ファイル。以下の多層防護を実装:

1. `.gitignore` に `.env` を追記 → git 管理外
2. `config_server.py` は `127.0.0.1` にのみバインド → ローカルのみ
3. デフォルト値は `PLACEHOLDER` → 誤って空のトークンで本番送信しない
4. 環境変数でも設定可能 (`os.environ` フォールバック) → CI/CD環境対応
